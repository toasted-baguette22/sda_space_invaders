Software Design Architecture Project - Space Invaders Refactoring

1. Application of Architectural Design Principles (SOLID)
=========================================================
The original Space Invaders project had several violations of the SOLID principles, particularly the Single Responsibility Principle (SRP) and the Dependency Inversion Principle (DIP). 
In the refactored code:
- **Single Responsibility Principle (SRP)**: The `SpacePanel` class originally managed all logic, rendering, and instantiation of objects. We extracted the instantiation logic into `EntityFactory` and image loading into `ResourceManager`. Collision reactions were decoupled from `SpacePanel` and moved to an event-based system using `GameObserver`.
- **Dependency Inversion Principle (DIP)**: `Player` and `Alien` classes previously depended on specific implementations (like `SpacePanel` directly) for notifying game-over states and collision handling. By introducing the `GameObserver` interface, the `Player` class now depends on an abstraction rather than a concrete panel.
- **Open/Closed Principle (OCP)**: The new structure makes the project easier to extend. If we add new entity types (like new power-ups or enemies), we simply extend `EntityFactory` without modifying the core `SpacePanel` game loop.

2. Implementation and Justification of Design Patterns
======================================================
Three core design patterns were implemented to address specific architectural issues in the codebase:

a) Singleton Pattern (`ResourceManager.java`)
- **Justification**: The original code instantiated `ImageIcon` and loaded image files directly in the constructors of `SpacePanel` and entities. This meant memory was wasted on multiple loads, and paths were hardcoded across multiple files.
- **Implementation**: The `ResourceManager` ensures only one instance manages the loading of graphical assets. All assets are loaded exactly once into a HashMap, providing global point of access and avoiding redundant I/O operations during game execution.

b) Factory Method Pattern (`EntityFactory.java`)
- **Justification**: Entity creation (Player, Alien, Laser, Bomb) was scattered throughout `SpacePanel`, `AlienHorde`, and `Alien`. This tightly coupled those classes to the concrete constructors of entities.
- **Implementation**: `EntityFactory` centralizes the creation logic. Whenever a new entity needs to be spawned, the factory provides it. This means `AlienHorde` no longer needs to pass around image objects just to instantiate an `Alien`. The Factory fetches the image from the `ResourceManager` and returns the complete entity.

c) Observer Pattern (`GameObserver.java`)
- **Justification**: `Player.java` had direct references to `SpacePanel` and `Lives` just to notify the game that a life was lost or the game was over. This created a strong two-way coupling.
- **Implementation**: `SpacePanel` now implements `GameObserver` (`onGameOver`, `onPlayerHit`). The `Player` only receives a `GameObserver` interface during collision checks. When a collision happens, it notifies the observer. This decouples the game engine GUI from the entity logic entirely.

3. Improvements in Maintainability and Scalability
==================================================
- **Maintainability**: Centralizing resource loading (`ResourceManager`) and object creation (`EntityFactory`) means that if asset paths change or if an entity's constructor signature changes, we only need to update the code in one location. 
- **Scalability**: The `GameObserver` interface sets the stage for a much larger system. If we want to add achievements, sound effects, or a networking module, we simply make them implement `GameObserver` and register them. We do not need to change the `Player` class or `SpacePanel` collision code. The factory pattern also allows easily introducing new enemy variants without modifying the horde management logic.
